# Journal App вЂ” Full Stack (Flask + React)

A full stack productivity tool where users can register, log in, and manage personal journal entries. The backend is a secure Flask REST API using server-side sessions. The frontend is a React app that handles authentication and proxies all API calls to Flask.

## How the Two Apps Connect

Three things work together to connect the frontend and backend:

**1. Proxy** вЂ” `client-with-sessions/package.json` contains:
```json
"proxy": "http://localhost:5555"
```
This means every `fetch("/signup")` call in the React app is automatically forwarded to `http://localhost:5555/signup`. You never need to write the full URL in frontend code.

**2. CORS** вЂ” The Flask backend uses `flask-cors` configured to accept requests from the React origin and allow session cookies to travel between them:
```python
CORS(app, origins=["http://localhost:3000"], supports_credentials=True)
```

**3. Session cookies** вЂ” Flask sets a signed cookie on the browser after login. The browser stores it and sends it automatically with every subsequent request, keeping the user logged in across page refreshes. These settings in `config.py` make the cookie work correctly across origins:
```python
SESSION_COOKIE_SAMESITE = "Lax"
SESSION_COOKIE_HTTPONLY = True
SESSION_COOKIE_SECURE   = False  # True in production over HTTPS
```

## Backend Setup

All commands below are run from inside the `server/` directory.

### Step 1 вЂ” Navigate to the server folder

```bash
cd flask-api-with-sessions-and-jwt-clients/server
```

### Step 2 вЂ” Install Python dependencies

```bash
pipenv install
```

This reads the `Pipfile` and installs all packages including Flask, SQLAlchemy, Bcrypt, Flask-Migrate, Flask-RESTful, Flask-CORS, and Faker into an isolated virtual environment.

### Step 3 вЂ” Activate the virtual environment

```bash
pipenv shell
```

Your terminal prompt will change to show `(server)` at the start. Every subsequent Python and Flask command must be run inside this shell.

### Step 4 вЂ” Create the environment file

```bash
cat > .env << 'EOF'
SECRET_KEY=super-secret-key-change-this
FLASK_APP=app.py
FLASK_ENV=development
EOF
```

The `SECRET_KEY` is used to sign session cookies. Change it to a long random string before deploying to production.

### Step 5 вЂ” Apply database migrations

```bash
flask db upgrade
```

This applies the migration files inside `migrations/versions/` and creates `instance/journal_app.db` with the `users` and `journal_entries` tables.

### Step 6 вЂ” Seed the database

```bash
python seed.py
```

This drops all existing data, recreates the tables, and inserts:
- 5 users with Faker-generated usernames and emails
- 40 journal entries spread across those users

All seed users share the password `password123`. Check the generated usernames with:

```bash
sqlite3 instance/journal_app.db "SELECT id, username FROM users;"
```

## Verifying the Connection

Once both servers are running, open `http://localhost:4000` and press `F12` to open **DevTools в†’ Network tab**. Work through each check in order.

### Check 1 вЂ” Page load fires check_session

The moment the page loads, `App.js` automatically sends:
```
GET /check_session   в†’   401
```
`401` means the route exists and correctly reports no active session. The login page should be visible. If you see `404` instead, the route is registered as `/me` in `app.py` вЂ” change it to `/check_session`.

### Check 2 вЂ” Sign up a new user

Click **Sign Up** and fill in the form:
```
Username:              journaluser
Password:              password123
Password Confirmation: password123
```

Network tab should show:
```
POST /signup   в†’   201
```

Response body:
```json
{ "id": 6, "username": "journaluser", "email": "" }
```

The login form disappears and **"You are logged in!"** appears. Open **Application в†’ Cookies в†’ localhost:4000** вЂ” a `session` cookie should now be present.

### Check 3 вЂ” Session persists on refresh

Hard refresh with `Ctrl + Shift + R`. Network tab should show:
```
GET /check_session   в†’   200
```
The app immediately shows the NavBar without the login form. The session cookie survived the refresh.

### Check 4 вЂ” Logout

Click **Logout**. Network tab should show:
```
DELETE /logout   в†’   204
```
The login form reappears. The session cookie is gone from Application в†’ Cookies.

### Check 5 вЂ” Login with correct credentials

Switch to **Log In** and enter:
```
Username:  journaluser
Password:  password123
```

Network tab should show:
```
POST /login   в†’   200
```

**"You are logged in!"** appears again.

### Check 6 вЂ” Login with wrong password

Logout first, then try:
```
Username:  journaluser
Password:  wrongpassword
```

Network tab should show:
```
POST /login   в†’   401
```

An error message appears on the form.

### Check 7 вЂ” Login as a seed user

Log out, then find a seed username:
```bash
sqlite3 instance/journal_app.db "SELECT username FROM users LIMIT 3;"
```

Log in with that username and `password123`. Should return `200`.

## Backend вЂ” Full Reference

### All API Endpoints

| Method | Endpoint | Auth required | Description |
|--------|----------|--------------|-------------|
| POST | `/signup` | No | Register a new user |
| POST | `/login` | No | Log in and start a session |
| DELETE | `/logout` | No | End the current session |
| GET | `/check_session` | No | Return the logged-in user |
| GET | `/entries` | Yes | Paginated list of own entries |
| POST | `/entries` | Yes | Create a new entry |
| PATCH | `/entries/<id>` | Yes | Update an entry (owner only) |
| DELETE | `/entries/<id>` | Yes | Delete an entry (owner only) |

### Endpoint details

#### `POST /signup`

```json
// Request body
{
  "username": "journaluser",
  "password": "password123",
  "password_confirmation": "password123"
}

// 201 Success
{ "id": 1, "username": "journaluser", "email": "" }

// 422 Errors
{ "errors": ["Username already taken."] }
{ "errors": ["Passwords do not match."] }
{ "errors": ["username and password are required."] }
```

#### `POST /login`

```json
// Request body
{ "username": "journaluser", "password": "password123" }

// 200 Success
{ "id": 1, "username": "journaluser", "email": "" }

// 401 Error
{ "errors": ["Invalid username or password."] }
```

#### `DELETE /logout`

No request body. Returns `204` with empty body.

#### `GET /check_session`

No request body.

```json
// 200 Success
{ "id": 1, "username": "journaluser", "email": "" }

// 401 Error
{ "errors": ["Not authenticated."] }
```

#### `GET /entries`

Query parameters: `page` (default 1), `per_page` (default 10, max 50).

```json
// 200 Success
{
  "entries": [
    {
      "id": 1,
      "title": "My first entry",
      "content": "Today was productive.",
      "mood": "happy",
      "is_private": true,
      "created_at": "2026-04-22T10:00:00",
      "updated_at": null,
      "user_id": 1
    }
  ],
  "total": 1,
  "pages": 1,
  "current_page": 1,
  "has_next": false,
  "has_prev": false
}
```

#### `POST /entries`

```json
// Request body вЂ” title and content required, mood and is_private optional
{
  "title": "My first entry",
  "content": "Today was productive.",
  "mood": "happy",
  "is_private": true
}

// 201 Success вЂ” returns the created entry object
// 401 вЂ” not logged in
// 422 вЂ” missing title or content
```

#### `PATCH /entries/<id>`

```json
// Request body вЂ” all fields optional
{ "title": "Updated title", "mood": "grateful" }

// 200 Success вЂ” returns updated entry object
// 401 вЂ” not logged in
// 403 вЂ” not the owner
// 404 вЂ” entry not found
```

#### `DELETE /entries/<id>`

No request body. Returns `204`, or `401`, `403`, `404`.

### Status codes

| Code | Meaning |
|------|---------|
| 200 | Success |
| 201 | Created |
| 204 | No content (logout, delete) |
| 401 | Not authenticated |
| 403 | Authenticated but not the owner |
| 404 | Resource not found |
| 422 | Validation error |

### Error format

All errors follow this structure:
```json
{ "errors": ["Human readable message."] }
```

### Running tests

```bash
cd server
pipenv shell
pytest tests/ -v
```

Tests use an in-memory SQLite database and Faker-generated data вЂ” they never touch `instance/journal_app.db`.

```bash

# List journal entries
sqlite3 instance/journal_app.db "SELECT id, user_id, title, mood FROM journal_entries LIMIT 10;"
```

## Frontend вЂ” Full Reference

### How each component talks to the backend

| Component | Method | Endpoint | Trigger |
|-----------|--------|----------|---------|
| `App.js` | GET | `/check_session` | Page load / refresh |
| `SignUpForm.js` | POST | `/signup` | Form submit |
| `LoginForm.js` | POST | `/login` | Form submit |
| `NavBar.js` | DELETE | `/logout` | Logout button click |

### Key behaviours

**Session check on load** вЂ” `App.js` fires `GET /check_session` every time the page loads. `200` shows the logged-in view. `401` shows the login page. This is what makes sessions survive page refreshes.

**Password confirmation** вЂ” `SignUpForm.js` sends `password_confirmation` alongside `password`. The backend compares them and returns an error if they differ.

**Error display** вЂ” Both forms expect errors as `{"errors": [...]}` вЂ” a list. They call `.map()` on `err.errors` to render each message. If the backend returns `{"error": "..."}` (singular) the frontend crashes with `Cannot read properties of undefined (reading 'map')`.

**No credentials flag needed** вЂ” Because the frontend uses the `proxy` setting in `package.json`, all fetch calls go through the same origin (`localhost:4000`) so the browser treats them as same-origin requests and sends cookies automatically. No `credentials: "include"` flag is needed in the fetch calls.

## Common Errors and Fixes

| Error | Where seen | Cause | Fix |
|-------|-----------|-------|-----|
| `GET /check_session в†’ 404` | Network tab | Route registered as `/me` | Change to `"/check_session"` in `app.py` |
| `Cannot read properties of undefined (reading 'map')` | Browser console | Backend returning `{"error": "..."}` not `{"errors": [...]}` | Update all returns in `auth_resources.py` to `{"errors": ["..."]}` |
| `Invalid email address` on signup | Browser form | Email validator rejecting empty string | Update `validate_email` in `models.py` to return `""` when value is falsy |
| CORS error in browser console | Browser console | `flask-cors` not configured | Add `CORS(app, origins=["http://localhost:3000"], supports_credentials=True)` to `app.py` |
| Logs out on every refresh | Browser | Session cookie not persisting | Add `SESSION_COOKIE_SAMESITE = "Lax"` to `config.py` |
| `Username already taken` on first signup | Browser form | Old curl test data still in DB | Run `python seed.py` to wipe and reseed |
| `flask: command not found` | Terminal | Virtual environment not active | Run `pipenv shell` first |
| `Module not found` on npm start | Terminal | Node modules not installed | Run `npm install` in `client-with-sessions/` |
| `404 Not Found` at `localhost:5555` | Browser | No route defined for `/` | Normal вЂ” test with `localhost:5555/check_session` instead |
| `Method Not Allowed` at `localhost:5555/signup` | Browser | Browser sends GET, route only accepts POST | Normal вЂ” use Postman or curl to test endpoints directly |
